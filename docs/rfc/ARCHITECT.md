# 总体架构说明

## 项目定位

WHUSingularity 的核心不是“做一个普通秒杀系统”，而是围绕分布式系统中的**单点热源问题**提出一套 `slot-based contention abstraction`。

对于一个会成为热点的单一 `resource`，项目不直接让所有请求竞争同一个中心状态，而是先把资源拆成多个 `slot`，再通过 `Registry -> ShardPolicy -> Interceptor -> Handler` 这条链路把请求分流到不同槽位，从而把单点争用转化为多槽并行争用。

在此基础上，上层以秒杀业务作为验证场景，进一步展示：

- 如何把热点治理抽象为框架能力，而不是写死在业务逻辑里
- 如何把一致性、吞吐、公平性之间的 trade-off 下沉到策略层
- 如何通过 MQ、Redis、幂等消费和自动扩缩容形成完整的工程闭环

## 单点高并发框架部分

### 核心接口

- `Slot`
- `Actor`
- `Allocator`
- `Registry`
- `ShardPolicy`
- `Interceptor`

这组接口抽象的是“多个 actor 并发竞争同一个热点 resource”的问题，而不是某一个具体业务接口。

其中：

- `Registry` 负责给出当前可用 slot 集合
- `ShardPolicy` 负责决定 actor 应该进入哪个 slot
- `Interceptor` 负责在分配过程中插入业务逻辑
- `Handler` 负责最终提交业务动作

这种设计把**热点分流、业务逻辑、一致性策略、性能治理**拆成了相对独立的层次。

### 业务微服务分离

- 用户服务（User）
- 商品服务（Product）
- 订单服务（Order）

## 课程汇报视角下的四个核心问题

下面四部分适合直接作为 pre / 答辩中的主线，每一部分都从“核心技术栈、创新点、优越性”三个角度展开。

### 1. 一致性

#### 核心技术栈

- `singularity-core`：`Registry + ShardPolicy + Interceptor + Handler`
- Redis Lua：在热路径中执行原子库存扣减与 shadow order 写入
- RocketMQ 事务消息：half message + commit / rollback
- Redis 回查：broker 回查本地事务状态
- MySQL + MQ Consumer：异步订单落库、异步库存落库

#### 创新点

- 框架本身**不硬编码某一种一致性模型**，而是把一致性要求留给业务策略层表达。
- `ShardPolicy` 不只是“哈希路由”，而是承载一致性偏好的策略入口：
- 如果业务更强调公平性，可以在 slot 集合变化时进行重哈希，使重试请求更容易路由到仍有余量的 slot。
- 如果业务更强调极限吞吐，也可以保持静态路由，让用户在其对应 slot 售罄后直接失败，用局部不公平换取系统稳定性。
- 换句话说，框架负责“分流和编排”，具体一致性等级由业务自行选择。

#### 优越性

- 相比把一致性逻辑写死在单个业务接口中，这种设计更适合适配不同业务场景。
- 相比“框架直接承诺强一致”，这种做法更符合工程现实，因为公平性、吞吐、重试语义本质上是业务 trade-off。
- 在订单链路中，系统能够形成一套清晰的正确性语义：
- Redis 决定抢单是否成立
- RocketMQ 决定消息是否被可靠持久化和传播
- MySQL 作为最终持久化结果承担异步落库职责

#### 汇报建议句式

可以直接说：

> 我的框架不直接绑定某一种一致性模型，而是提供一个可插拔的策略空间，让业务在公平性优先、吞吐优先、事务可靠优先之间自行权衡。

### 2. 错误处理

#### 核心技术栈

- RocketMQ 事务消息与 broker 回查
- Redis shadow order 记录
- 订单服务消费端幂等写库
- 库存服务消费端 `messageId` 幂等校验与变更日志
- 客户端透明重试 + 服务端补偿回滚

#### 创新点

- 项目对“可靠投递”的核心理解不是堆很多容错名词，而是抓住两条基本原则：
- 要么请求是透明转发的，失败后可以无损重试
- 要么中转节点必须持久化，保证中途失败也不丢信息
- 在订单场景里，MQ 就是持久化中转点，因此系统把“成功语义”锚定在事务消息和回查机制上，而不是锚定在某个瞬时进程状态上。
- 项目的错误处理重点不是熔断降级，而是**重试补偿 + 幂等消费 + 可恢复链路**。

#### 优越性

- 相比把错误处理做成大量复杂的控制逻辑，这套方案更适合秒杀这种写热点场景，因为重点是“不丢消息”和“能够恢复”。
- Redis 写入失败、MQ 发送失败、消费者重复消费、进程中途崩溃，这几类核心失败场景都有明确恢复路径。
- 订单侧与库存侧都通过幂等设计降低了“至少一次投递”带来的重复副作用。

#### 汇报建议句式

可以直接说：

> 我们的错误处理核心不是熔断，而是保证失败请求可恢复、消息不丢失、消费结果幂等化，因此系统的重点是重试补偿而不是复杂降级。

### 3. 性能处理

#### 核心技术栈

- `slot` 分桶 + `ShardPolicy` 路由
- Redis 多 key / 多 bucket 热点分散
- `singularity-order-baseline` 对照组
- `Interceptor` 链承载业务计算逻辑
- `singularity-scaler`：CPU / 内存 / QPS 指标驱动的自动扩缩容
- k6 压测脚本与业务成功率口径

#### 创新点

- 项目不是简单说“Redis 很快”，而是先识别出真正的性能瓶颈是**单点热 key / 中心状态源**。
- 因此，性能优化首先不是数据库调优，而是把单一热点 resource 拆成多个 slot，降低单点争用。
- 同时，项目引入了 baseline 对照组，证明框架价值不在“单机一定更快”，而在“扩展曲线是否还能继续增长”。
- 对无状态服务而言，如果业务逻辑太轻，扩实例并不能提升总吞吐，因为最终还是卡在 Redis 或 MQ 上。
- 因此项目把风控、用户资质、规则匹配等真实业务计算以 `Interceptor` 形式嵌入链路，使服务具备可横向扩展的 CPU 型负载，从而真正体现 scaler 的价值。

#### 优越性

- 相比单 Redis 热 key 方案，slot 分流可以显著降低中心化串行化压力。
- 相比“只会堆机器”的扩容方式，这个框架先处理热点结构，再做横向扩展，因此扩容是有收益的。
- 相比只做优化版而没有对照组，这个项目因为存在 baseline，所以能更清楚地回答：
- 单机吞吐谁更高
- 为什么 baseline 在低负载下可能更快
- 为什么优化版在高负载、多实例下更有扩展性
- 这样在课程汇报里，你讲的不再是“调了参数”，而是“解释吞吐曲线为何变化”。

#### 汇报建议句式

可以直接说：

> 这个框架的价值不只是提高单机性能，而是把一个不可扩展的热点写链路改造成可以随着实例数继续增长的系统。

### 4. 开发友好度

#### 核心技术栈

- Spring IOC / Spring Boot 自动装配能力
- `Interceptor` 链式扩展点
- `Context` 传递业务上下文
- DDD 或传统 MVC 都可接入
- Nacos 统一配置 slot 元数据与服务配置

#### 创新点

- 项目把热点治理做成了一个低侵入框架，而不是要求业务方大规模重写代码。
- 业务方只需要关注：
- 如何定义 slot
- 如何定义路由策略
- 哪些横切逻辑放进 interceptor
- 最终 handler 如何提交业务动作
- 例如新增风控或 CPU 型业务逻辑时，不需要重写整条秒杀链路，只需要新增一个 interceptor 即可。
- 这使框架既能适配 DDD 风格，也能适配传统 MVC 风格。

#### 优越性

- 相比把业务判断、日志、风控、限流全部堆在一个 Service 方法里，这种结构更清晰，也更利于后期维护。
- 相比强绑定某个具体业务模型，这个框架抽象层级更高，因此具备复用潜力。
- 对课程设计来说，这一点非常重要，因为它说明项目不仅有性能收益，也有工程组织能力上的收益。

#### 汇报建议句式

可以直接说：

> 我们不仅追求更高吞吐，也追求更低的业务侵入性；新增一类治理逻辑时，通常只需要新增一个 interceptor，而不是改写整条业务链路。

## 适合 PPT 首页或结尾页的总结

如果要把整个项目压缩成几句话，建议使用下面这组表达：

- 本项目的核心创新不是中间件堆叠，而是提出了针对分布式单点热源问题的 `slot` 分流抽象。
- 框架通过 `Registry / ShardPolicy / Interceptor / Handler` 把热点分流、业务逻辑和一致性策略解耦。
- 在订单链路中，项目进一步结合 Redis、RocketMQ、幂等消费和 MySQL 异步落库，形成了可恢复的可靠写链路。
- 在性能层面，项目不仅有优化版，还有 baseline 对照组和 scaler 扩缩容闭环，因此能够说明“为什么系统可扩展”，而不是只展示某次压测结果。
- 在工程层面，框架支持 Spring 生态、低侵入接入和横切逻辑插拔，兼顾性能收益与开发友好度。

## 其他创新点

- 可进一步扩展更复杂的 ShardPolicy，以表达公平性优先、吞吐优先、重试友好等不同业务偏好。
- 可继续把 slot 从“库存桶”升级为“业务规则单元”，承载更多元数据，如风控阈值、用户等级限制、优惠规则等。
- 可结合 WebMCP / Agent 能力继续向更智能的运维、观测与控制面演进。
