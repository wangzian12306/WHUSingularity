# 用户系统鉴权设计（A档）- 概览与边界

> 版本：v1.0
> 日期：2026-04-09
> 适用范围：`WHUSingularity` 当前三服务形态（`user` / `order` / `eureka`），无网关部署

## 1. 背景

`WHUSingularity` 是“高并发抢单框架 + Spring Cloud 微服务实践”项目。业务上至少包含用户服务、订单服务与服务注册中心。抢单链路强调高并发与异步一致性（Redis + MQ），因此登录注册设计必须能支持跨服务身份识别，而不只是单服务登录。

## 2. 目标（A档：课程可落地）

本档目标只覆盖“最小闭环”，优先可实现、可联调、可验收：

1. 新用户可注册并登录。
2. 登录后可携带令牌访问抢单接口。
3. 退出后当前令牌立即失效。
4. 用户敏感信息不外泄（尤其密码）。
5. 错误语义统一（401/403/409/400）。

## 3. 非目标（本档不做）

1. 多因子认证（MFA）。
2. 第三方登录（OAuth/微信/手机号验证码）。
3. 刷新令牌全套体系（可后续扩展）。
4. 单点登录（SSO）与跨端会话编排。
5. 风控引擎（设备指纹、IP信誉、行为分析）。

## 4. 服务职责边界

| 组件 | 职责 | 不负责 |
|---|---|---|
| 前端 | 调用注册/登录/退出接口；保存并携带 `Bearer token`；收到 401 引导重登 | 自己计算/伪造用户身份 |
| user 服务 | 账号注册、密码校验、签发 token、注销失效、返回当前登录用户信息 | 抢单库存逻辑、订单落库 |
| order 服务 | 校验 token，提取 `userId/role`，在此基础上执行抢单 | 信任请求体内 `userId` |
| Auth Layer（部署在 user/order 服务内部） | 统一完成 token 解析、签名校验、过期校验、黑名单校验，并把 `userId/role` 注入请求上下文 | 在 controller 中分散编写鉴权逻辑 |
| Redis | token 失效黑名单、必要缓存 | 长久化用户主数据 |
| MySQL | 用户账号、密码哈希、角色等主数据存储 | token 在线会话状态（A档） |

## 5. 核心设计原则

1. **身份来自 token，不来自请求体**：order 侧一律从 token 取 `userId`。
2. **最小暴露**：接口返回用户公开字段，不返回 `password`。
3. **退出可感知**：logout 后旧 token 立即不可用。
4. **薄 Auth Layer**：在各服务内用统一过滤器/拦截器承载鉴权，不在 controller/service 里散落校验代码。
5. **先无网关、后可迁移**：Auth Layer 组件化封装，未来可上移到网关。

## 6. 端到端主流程（总览）

1. **注册**：`register` 创建用户（密码哈希），默认 `role=normal`。
2. **登录**：`login` 校验密码，返回 access token（含 `sub/role/jti/exp`）。
3. **鉴权入口（Auth Layer）**：受保护请求先经过 Auth Layer，完成 token 解析、签名/过期检查、黑名单检查，并将 `userId/role` 注入上下文。
4. **抢单调用**：前端带 `Authorization: Bearer <token>` 调用 order 抢单接口；order 的 Auth Layer 验证通过后进入业务链路。
5. **退出**：`logout` 将 token 对应 `jti` 写入 Redis 黑名单；后续请求命中黑名单返回 401。

## 7. 与当前代码状态的关系（截至 2026-04-09）

当前阶段进展如下：

1. user 服务已完成注册/登录契约收口，响应脱敏，不再在主鉴权接口中返回 `password`。
2. user 服务已完成 BCrypt 密码哈希存储与登录校验。
3. user 服务已完成 JWT 签发与校验（`sub/role/jti/iat/exp`）以及 Redis 黑名单失效。
4. user 服务已完成 `logout`、`me` 与受保护接口鉴权入口（Filter 方案）。
5. user 服务已补充 `admin` 受保护接口示例与 403 鉴权语义验证。

## 8. 迁移到网关的兼容策略

未来若新增网关：

1. 将“token 验签 + 黑名单检查”从各业务服务迁移到网关。
2. 网关把解析后的身份透传给下游（如请求头注入 `X-User-Id`/`X-User-Role`，并在内网可信边界内使用）。
3. 业务服务保留兜底校验能力（灰度期）。
