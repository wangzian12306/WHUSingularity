## Q1：秒杀系统的痛点是什么？（0:00 – 0:20）

秒杀的本质是：**瞬时请求量远超服务器正常处理能力**。
单机扛不住，数据库锁争抢严重，库存超卖，系统崩溃。

> *这是个纯粹的资源稀缺 + 高并发竞争问题。*

**PPT 建议**：一张"流量脉冲 vs 服务器处理线"的示意图，直观呈现峰值压力。

---

## Q2：加机器就能解决吗？（0:20 – 0:40）

加机器是对的方向——**横向扩展（Scale Out）**。
但不是加了机器就完事，随之而来的问题是：

> **多台机器怎么协作？状态放在哪里？谁来调度？**

这就是我们这个项目真正要回答的问题。

---

## Q3：横向扩展的前提是什么？（0:40 – 1:10）

横向扩展能成立，有一个核心前提：**服务必须是弱状态的**。

强状态（把用户会话、库存计数都存在本地内存）的服务加再多机器也没用——
每台机器看到的世界不一样，结果不一致。

弱状态意味着：**本地不保存需要跨机器共享的关键状态，或者业务可以接受一定程度的最终一致性**。
状态下沉到 Redis、MySQL、MQ 这些共享基础设施。

> *这是 singularity-core 框架设计的出发点。*

---

## Q4：弱状态下，如何抽象资源分配？（1:10 – 2:00）

我们把"高并发资源争抢"提炼成一个通用模型：

- **Slot（资源桶）**：系统里有限的资源被切成若干桶，每桶有自己的状态（如 Redis 计数器）；
- **Actor（请求者）**：每个进来的请求携带身份和元数据；
- **Allocator**：唯一入口，`allocate(Actor) → Result`。

执行流程：

```
Actor 进入
  → Registry 过滤掉已耗尽的 Slot，返回可用列表
    → ShardPolicy 把 Actor 路由到某一个 Slot（哈希，同一用户总打同一桶）
      → Interceptor 链执行业务逻辑
        → 返回 Result
```

**关键设计决策**：Allocator 不决定"分配之后怎么改变状态"，那是业务的事。
框架只负责**把正确的请求送到正确的桶**。

**PPT 建议**：Actor → Registry → ShardPolicy → Slot 的箭头流程图，旁边注明各自职责。

## Q5：分配逻辑如何做到可插拔、不侵入业务？（2:00 – 2:40）

答案是 **Interceptor 链**，around 语义：

```
[风控拦截器]  →  [日志拦截器]  →  [业务 Handler]
     ↑                ↑                ↑
  pre-logic        pre-logic        执行业务
  next() →         next() →
  post-logic       post-logic
```

每个拦截器调用 `ctx.next()` 把控制权传给下一个，最终回到自己做后置处理。
风控、日志、限流、降级——全部作为独立 Interceptor 插入链条，**业务 Handler 里零感知**。

具体在秒杀场景中，我们插入了一个 **FraudDetectionInterceptor**：
计算每个用户的 4 维行为特征哈希（请求频率、时间窗口、Slot 偏好、跨桶切换），
风险分超过阈值直接拦截，不进入库存扣减流程。

> *同时，这个拦截器有意设计为 CPU 密集型——它是我们触发 Scaler 扩容的"压力制造者"，后面会讲到。*

## Q6：秒杀场景下，Slot 里的"状态"具体怎么放？（2:40 – 3:20）

每个 `StockSlot` 绑定一个 **Redis 计数器**（如 `stock:bucket-1`）。

- 库存分散到多个桶，哈希分流让并发压力分摊；
- 当某个桶计数器归零，Registry 实时把它从可用列表里剔除，后续请求不再打过来；
- 这样避免了"全部请求挤一把全局锁"的问题。

**PPT 建议**：多个 Slot 桶各自绑定 Redis key 的示意图，对比"单一全局库存"方案。

## Q7：库存扣减和订单写入，如何保证不出现超卖或数据丢失？（3:20 – 3:50）

这是分布式事务问题。我们用 **RocketMQ 半消息 + Redis Lua 脚本** 解决：

1. 先向 Broker 发一条**半消息**（消费者暂时看不见）；
2. 执行 Lua 脚本**原子扣减** Redis 库存——成功则提交半消息，失败则回滚；
3. 消费者收到消息后，分别完成订单落库（Order 服务）和 MySQL 库存扣减（Stock 服务）；
4. 如果进程在步骤 2-3 之间崩溃，Broker 主动**回查** Redis，根据 key 是否存在决定补偿提交还是丢弃。

> *整个链路不需要两阶段提交，靠消息队列实现最终一致性。*

---

## Q8：框架解决了单次请求的调度，但实例数谁来管？（3:50 – 4:10）

框架本身不管实例数——那不是它的职责。
但流量高峰来了，一台机器的 CPU 被打满，再好的框架也没用。

> **我们需要一个能感知压力、自动增减实例数的服务——这就是 singularity-scaler。**

---

## Q9：Scaler 凭什么知道该扩容了？（4:10 – 4:50）

Scaler 同时从**两路**采集指标，互为补充：

| 来源 | 采集方式 | 作用 |
|---|---|---|
| **JVM / Prometheus** | HTTP 拉取 `/actuator/prometheus` | CPU 使用率、内存、QPS 增量 |
| **Docker Stats** | `docker stats --no-stream` | 宿主机原始资源，JVM 崩溃时仍有效 |

为什么要两路？
单靠 JVM 指标：JVM 崩了就没数据，扩容失效；
单靠 Docker Stats：JVM 启动时 CPU 天然峰值，容易误触发。
**两路交叉验证，才能做出可信决策。**

扩容触发条件：任意一路 CPU 超过 70%，或 15 秒内 CPU 急升超过 38 个百分点，立即加一个实例。

---

## Q10：缩容为什么比扩容谨慎得多？（4:50 – 5:20）

扩容宁可多扩，代价是多花一点资源；缩容缩错了，直接影响服务可用性。

所以缩容有三道门槛：

1. **连续 3 个采样周期**（约 45 秒）所有指标都低于下限，不是偶发低谷；
2. **Docker CPU 也必须低于阻断阈值**——JVM 指标可能因为 GC 暂停而虚低，Docker 数据不说谎；
3. **新容器有 90 秒冷静期**——JVM 启动时 CPU 天然高，这段时间的数据不参与缩容判断。

> *扩容快、缩容慢——这是有意为之的非对称设计。*

---

## 收尾：闭环总结（5:20 – 5:40）

整个系统形成一个完整的闭环：

```
高并发请求涌入
  → singularity-core 分槽路由 + Interceptor 链有序处理
    → FraudDetectionInterceptor 制造 CPU 压力（故意的）
      → Scaler 感知压力，docker compose --scale 加实例
        → 新实例注册 Nacos，Gateway 自动纳入负载均衡
          → 压力下降，Scaler 稳定后缩容
```

**一句话收尾**：
singularity-core 和 scaler 的设计都是通用的，秒杀只是我们选择的极端验证场景。
任何需要横向扩展资源分配的系统，都可以套用这套框架。

- Q7（分布式事务）可以根据时间压缩为一句话："RocketMQ 半消息 + Redis 原子操作，Broker 负责崩溃补偿"，不展开细节。
