## 一、项目定位

**WHUSingularity** 是一个面向电商秒杀场景的分布式微服务系统，核心解决高并发抢单场景下的 **库存超卖、锁竞争、数据一致性** 三大难题。

| 维度 | 说明 |
|------|------|
| 业务场景 | 电商秒杀/抢购：商户上架限量商品 → 大量用户同时抢购 → 订单支付 |
| 核心挑战 | 瞬时高并发请求、库存扣减原子性、分布式事务一致性、服务弹性伸缩 |
| 解决思路 | Actor-Slot 分槽路由 + Redis Lua 原子减库存 + RocketMQ 事务消息 + 两级缓存 |

## 四、核心创新点

### 4.1 自研 Actor-Slot 分槽路由模型（singularity-core）

**问题**：传统秒杀系统所有请求竞争同一行库存记录，数据库行锁成为瓶颈。

**方案**：将商品总库存分割为多个独立 Slot（库存桶），每个 Slot 拥有独立的 Redis Key 和锁空间。

```
传统方案：                          Actor-Slot 方案：
10000 个请求 → 1 个库存行 → 行锁   10000 个请求 → Hash 分片 → N 个 Slot → 各自独立扣减
吞吐量：受限于单行锁               吞吐量：线性扩展（Slot 数 × 单 Slot 吞吐）
```

**架构层次**：

| 抽象 | 接口 | 作用 |
|------|------|------|
| Actor | `Actor` | 请求发起者（用户），携带 ID 用于哈希分片 |
| Slot | `Slot` | 库存槽位，持有 ID + 元数据（redisKey, productId） |
| ShardPolicy | `ShardPolicy` | 分片策略接口，默认 HashShardPolicy |
| Allocator | `Allocator` | 分配器，协调 Actor → Slot 路由 + 拦截器链执行 |
| Interceptor | `Interceptor` | 拦截器，可插拔的业务切面 |

**管道式拦截器链**（Pipeline）：

```
Request → RateLimit → CacheStampedeGuard → Routing → Degrade → Handler → Response
           限流         缓存击穿防护         路由      降级      业务
```

| 拦截器 | 功能 |
|--------|------|
| `ReadRateLimitInterceptor` | 滑动窗口限流，防止突发流量击穿 |
| `CacheStampedeGuardInterceptor` | 缓存击穿防护，加锁 + 等待回源 |
| `ReadRoutingInterceptor` | 按 ShardPolicy 路由到目标 Slot |
| `ReadDegradeInterceptor` | 降级兜底，异常时返回 Fallback |
| `MetricsTraceInterceptor` | 可观测性埋点 |

### 4.2 Redis Lua 原子减库存 + 事务消息

**抢单核心流程**（三步显式事务）：

```
┌──────────────────────────────────────────────────────────┐
│  Step 1: 发送半消息 (Half Message)                       │
│  Order → RocketMQ: "order-topic" 半消息                  │
│  Listener 返回 UNKNOW（暂不提交）                         │
├──────────────────────────────────────────────────────────┤
│  Step 2: 执行本地事务 (Redis Lua)                        │
│  ┌──────────────── Lua 脚本 ────────────────┐            │
│  │ stock = GET key                           │            │
│  │ if stock <= 0 then return -1              │  ← 库存不足│
│  │ remaining = DECR key                      │            │
│  │ if remaining < 0 then INCR; return -1     │  ← 超卖防护│
│  │ HSET order:xxx orderId/actorId/status...  │  ← 写影子单│
│  │ return remaining                          │            │
│  └───────────────────────────────────────────┘            │
│  原子性：单条 Lua 在 Redis 中串行执行，无竞态             │
├──────────────────────────────────────────────────────────┤
│  Step 3: 提交或回滚                                      │
│  Lua 成功 → COMMIT_MESSAGE → 消费者持久化订单 + 扣减库存  │
│  Lua 失败 → ROLLBACK_MESSAGE → 半消息丢弃                │
└──────────────────────────────────────────────────────────┘

异常兜底：Broker 回查 → checkLocalTransaction → 查 Redis 影子单 → 判定提交/回滚
```

**优越性对比**：

| 维度 | 传统方案 | 本项目方案 |
|------|---------|-----------|
| 库存扣减 | MySQL `UPDATE stock SET count=count-1` 行锁 | Redis Lua 原子减，单线程无锁 |
| 超卖防护 | 乐观锁/悲观锁 | Lua 脚本内 `DECR + <0 回滚` |
| 数据一致性 | 本地事务 | RocketMQ 事务消息 + 影子单回查 |
| 故障恢复 | 依赖 DB 事务日志 | Broker 回查 Redis 影子单自动恢复 |

### 4.3 两级缓存架构（Caffeine + Redis）

**商品服务**采用 Cache-Aside 模式 + 两级缓存：

```
请求 → Caffeine 本地缓存 (5min TTL)
         │ miss
         ↓
       Redis 远程缓存 (详情30min / 列表10min)
         │ miss
         ↓
       MySQL 数据库
         │
         ↓ 回写
       Caffeine + Redis 双写
```

| 防护机制 | 说明 |
|---------|------|
| **缓存穿透** | NULL 占位符（`__NULL__`），空结果缓存 60s |
| **缓存击穿** | Redis 分布式锁 + 单飞模式，仅一个请求回源 |
| **缓存雪崩** | TTL 随机抖动，Caffeine 本地缓存兜底 |
| **缓存一致性** | 写操作时双删（本地 + Redis），列表全量失效 |

### 4.4 自动扩缩容（singularity-scaler）

| 能力 | 说明 |
|------|------|
| 指标采集 | Prometheus Actuator 端点，CPU/内存/QPS |
| 决策引擎 | 滑动窗口平滑 + 阈值判定 + 冷却期 |
| 执行动作 | Docker Compose `--scale` 增减实例 |
| 目标服务 | Order（核心抢单路径）优先扩容 |

## 五、分布式设计亮点

### 5.1 服务拆分原则

| 原则 | 体现 |
|------|------|
| **数据库独立** | 每个服务拥有独立 MySQL 库（user/order/stock/product/merchant） |
| **认证独立** | 用户 JWT 与商户 JWT 双体系，互不干扰 |
| **异步解耦** | Order → RocketMQ → Stock，抢单与库存持久化异步化 |
| **快速路径** | 抢单热路径只走 Redis，MySQL 写入由 MQ 消费者异步完成 |

### 5.2 容错与高可用

| 机制 | 说明 |
|------|------|
| **事务消息回查** | 进程崩溃后 Broker 回查 Redis 影子单，自动恢复 |
| **Slot 售罄标记** | 库存归零时标记 Slot 为 empty，后续请求快速失败 |
| **Nacos 健康检查** | 服务实例自动注册/摘除，Gateway 路由自动更新 |
| **Nginx 多副本 LB** | 3 台 Nginx 分散负载均衡单点，宿主多端口暴露 |
| **Gateway 负载均衡** | Spring Cloud LoadBalancer 客户端侧负载均衡 |

### 5.3 前后端分离架构

```
前端 (Vite Dev Server :5173)
  │
  ├── /api/* → Vite Proxy → Gateway (:8080)
  │                                │
  │                    ┌───────────┼───────────┐
  │                    ↓           ↓           ↓
  │              Nacos 路由    负载均衡    服务实例
  │
  ├── JWT Token 自动注入（用户/商户双 Token）
  ├── 401 自动清除 Token + 跳转登录
  └── 路由守卫（AuthGuard / AdminGuard）
```

## 七、项目优越性总结

### 7.1 vs 传统单体秒杀

| 对比维度 | 传统单体 | WHUSingularity |
|---------|---------|----------------|
| 架构 | 单体应用，耦合严重 | 6 个微服务 + 网关，独立部署 |
| 库存扣减 | MySQL 行锁，QPS 受限 | Redis Lua 原子操作，10x+ 吞吐 |
| 超卖风险 | 乐观锁/悲观锁，极端情况仍可能超卖 | Lua 脚本原子判定，零超卖 |
| 数据一致性 | 本地事务，跨服务无法保证 | RocketMQ 事务消息 + 影子单回查 |
| 扩展性 | 垂直扩展，天花板明显 | Actor-Slot 水平分片 + 自动扩缩容 |
| 缓存 | 单层 Redis 或无缓存 | Caffeine + Redis 两级缓存 + 击穿/穿透/雪崩防护 |
| 容错 | 进程崩溃 = 数据丢失 | Broker 回查 + 影子单自动恢复 |

### 7.2 核心竞争力

1. **自研 singularity-core 框架**：Actor-Slot 分槽路由 + 管道式拦截器链，可插拔、可扩展，非简单 Demo 级实现
2. **零超卖保证**：Redis Lua 原子减库存 + 事务消息三步提交，数学层面杜绝超卖
3. **高可用设计**：Nacos 服务发现 + Gateway 负载均衡 + Nginx 多副本 LB + 自动扩缩容
4. **完整业务闭环**：从商户上架 → 用户浏览 → 抢单 → 支付 → 管理后台，全链路可运行
5. **容器化全栈部署**：Docker Compose 一键启动 11 个容器，含压测工具 k6
6. **前沿技术选型**：Java 21 + Spring Boot 3.2 + Spring Cloud 2023 + Spring Cloud Alibaba 2023

### 7.3 技术深度指标

| 指标 | 说明 |
|------|------|
| 微服务数 | 6 个核心服务 + 1 个网关 + 1 个自动扩缩容 |
| 数据库隔离 | 5 个独立 MySQL 库 |
| 缓存层级 | 3 级（Caffeine 本地 → Redis 远程 → MySQL 持久化） |
| 消息可靠性 | 事务消息 + 半消息 + Broker 回查 |
| 库存分片 | Slot 数可配置，线性扩展吞吐 |
| 压测能力 | k6 渐进式 RPS（800→2400），多副本 + 多 LB 并行 |
